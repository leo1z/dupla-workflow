Tokens

Dónde van los tokens — antes y después

La diferencia entre explicar el contexto manualmente y usar el SESSION block.

Explicar el proyecto ~800

✅ Con Dupla Workflow

El sistema carga solo el SESSION block (~60 tokens) para conocer el estado. El resto (ROADMAP, ARCHITECTURE, PROBLEMS) se carga condicionalmente — solo cuando la tarea lo requiere. Los skills no consumen tokens hasta que los invocas.

Qué hace /checkpoint que no hace git
Git guarda el código. /checkpoint guarda el estado estratégico: en qué fase estás, qué salió, qué sigue, si alcanzaste un GO/NO-GO. Sin checkpoint, la próxima sesión reconstruye el estado del código — pero pierde la sincronización con el ROADMAP.
Inteligencia

Carga condicional — el sistema decide qué leer

/new-session detecta automáticamente qué docs necesitas según lo que vas a hacer. No tienes que pedírselo.

Si el campo "Next" contiene… El sistema carga… Tokens extra
plan decide fase roadmap evaluar ROADMAP.md completo (fases adyacentes también) +300–800
implement build crear conectar ARCHITECTURE.md (sección relevante) +400–1000
fix error bug roto PROBLEMS.md +200–500
Ninguna de las anteriores Nada extra — SESSION block es suficiente +0
Lo mismo aplica a /checkpoint
/checkpoint usa señales del git diff y mensajes de commit para decidir si preguntar sobre ARCHITECTURE o PROBLEMS. Si los archivos cambiados no incluyen schemas ni configs, no pregunta nada. Si los commits dicen "fix" o "error", ofrece agregar a PROBLEMS.md. Siempre signal-based — nunca pregunta por default.
Conexiones

Cómo se conectan los documentos entre sí

Cada documento tiene un rol único. Ninguno duplica información del otro.

📋
PROJECT_STATE.md ⭐ — fuente de verdad operacional
El bloque <session> contiene: qué se hizo, qué sigue, qué bloquea, en qué fase estás. Es lo único que /new-session lee por default. /checkpoint lo actualiza. Los hooks lo vigilan. Todo lo demás apunta a él.

Conectado con: ROADMAP (sincroniza fase), git log (reconstruye Done), CLAUDE.md (define cómo actuar).
🗺️
ROADMAP.md — brújula estratégica
Define qué significa "terminado" para cada fase. Fases con Outcomes (entregables concretos) y criterios GO/NO-GO (continúa si… / mata si…). /checkpoint lo marca cuando completas outcomes. /new-session lo lee para saber si estás en camino o si hay una decisión pendiente.

NO es: lista de tareas diarias — eso va en PROJECT_STATE. El ROADMAP es el plan estratégico; el SESSION block es el estado táctico.
🏗️
ARCHITECTURE.md — memoria técnica
Decisiones de stack, diagramas, "¿por qué usamos X en lugar de Y?". Se actualiza solo cuando /checkpoint detecta cambios en archivos de arquitectura (schemas, configs, package.json). No se toca en cada sesión — solo cuando algo arquitectónico cambió.

Conectado con: PROJECT_STATE (referenciado desde Next si construyendo), ROADMAP (outputs técnicos por fase).
⚙️
CLAUDE.md — contexto de comportamiento
Global (~/.claude/CLAUDE.md): identidad, reglas de trabajo, model routing. Proyecto (./CLAUDE.md): cómo el stack se conecta, reglas específicas del proyecto ("no tocar la DB en esta etapa"), dónde están los archivos críticos. Siempre cargado — no requiere invocación.

Cargado automáticamente por Claude Code en cada sesión. Define cómo actuar — los otros documentos definen dónde estás.

¿Cuándo actualizarlo? CLAUDE.md es configuración — no cambia en cada sesión. Si el stack cambia significativamente o las reglas del proyecto evolucionan, corre /adapt-project: detecta el drift entre el código actual y CLAUDE.md, y propone actualizaciones con diff y aprobación. No se auto-actualiza en checkpoint.
Automación

Hooks — automatizaciones fuera del contexto

Los hooks son shell scripts que Claude Code ejecuta en respuesta a eventos del sistema. Corren fuera de la ventana de contexto — no queman tokens.

⏰
session-reminder.sh
Cuándo: Al primer mensaje de cada sesión (UserPromptSubmit)

Qué hace: Lee PROJECT_STATE.md. Si el estado tiene más de 24h, avisa una sola vez: "Estado de hace Xd. Corre /new-session para actualizar."

Optimización: Usa un marker .tmp/session-reminder-fired — solo dispara una vez por sesión.
💾
suggest-checkpoint.sh
Cuándo: Al finalizar cada respuesta (Stop event)

Qué hace: Cuenta commits sin guardar. Si hay más de 5 commits sin checkpoint: avisa. Si hay cambios y han pasado 3 respuestas: sugiere /checkpoint.

Optimización: Throttle vía contador — no interrumpe en cada respuesta, solo cuando hay señal real.
🛡️
guard-project-state.sh
Cuándo: Antes de cualquier escritura (PreToolUse Write)

Qué hace: Verifica que PROJECT_STATE.md exista antes de que Claude escriba archivos del proyecto. Silencioso si todo está bien — solo bloquea si el archivo crítico está ausente.

Optimización: Completamente silencioso en el 99% de los casos.
Por qué los hooks no cuestan tokens
Claude Code ejecuta los hooks como procesos de sistema separados, antes o después de las operaciones del modelo. El output del hook aparece como contexto del sistema — no como parte del contexto del LLM. Es la razón por la que pueden monitorear sin "hablar" en cada turno.

Cómo escalar de micro a completo
Si el proyecto micro crece y necesita más estructura, corre /adapt-project en la misma carpeta. El sistema detecta el QUICKSTATE.md, lo usa como contexto inicial, y genera el conjunto completo de docs/ sin perder el historial.
Handoff

Cambiar de modelo — Claude ↔ Gemini

El estado persiste entre modelos. Un LLM puede continuar exactamente donde el otro dejó.

1. Generar handoff
Corre /checkpoint → elige opción 3 (Handoff). El sistema genera un bloque XML con: fecha, origen, destino, goal, branch, lo que se hizo, y qué sigue.
2. Persistir en PROJECT_STATE
El bloque <handoff> se escribe automáticamente en PROJECT_STATE.md justo después del bloque <session>. Si ya existía uno, lo reemplaza.
3. El siguiente modelo lo lee
El nuevo LLM corre /new-session. Detecta el bloque <handoff> en PROJECT_STATE.md y lo usa como contexto antes del SESSION block. Puede continuar sin que el usuario re-explique nada.
¿Por qué escribirlo al archivo?
Si el chat actual se cierra antes de que el handoff se complete, la información se pierde. Escribiendo al archivo, el estado sobrevive aunque el chat muera. El próximo LLM lee PROJECT_STATE.md — siempre disponible.
