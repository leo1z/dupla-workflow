Concepto

El sistema en una idea

Dupla OS es un arnés de markdown. El workflow vive en archivos .md — no en el LLM ni en el IDE. Conectas cualquier IA como "cerebro" y el sistema funciona igual: contexto, estado, memoria, skills.

📄
Archivos .md como estado
PROJECT_STATE.md es la memoria. ROADMAP.md es el plan. El LLM los lee al inicio de cada sesión — sin que tengas que re-explicar nada.
🧠
Skills como programas de prompt
Cada skill es un archivo .md con instrucciones. El LLM lo ejecuta cuando escribes /comando. Cero código, cero dependencias.
⚙️
Hooks como automatizaciones
Scripts bash que corren fuera del contexto — sin quemar tokens. Avisan cuando hay que hacer checkpoint, guardan snapshots, sincronizan entre modelos.
# Estructura de un proyecto Dupla OS
mi-proyecto/
├── CLAUDE.md                # Contexto del proyecto (lo que el LLM sabe siempre)
├── claude-progress.txt      # Estado ligero — leído en ~30 tokens
├── docs/
│   ├── PROJECT_STATE.md     # Fuente de verdad: dónde estás, qué sigue, qué bloquea
│   ├── ROADMAP.md           # Fases + GO/NO-GO checkpoints (dinámico)
│   ├── ARCHITECTURE.md      # Decisiones técnicas (solo si es software)
│   ├── SPEC.md              # Problema · Usuario · MVP · Kill condition
│   └── PROBLEMS.md          # Bugs y soluciones reusables
└── .agents/rules/claude.md  # Copia para Gemini/Antigravity
Por qué markdown y no una base de datos o app
Los archivos .md son universales — cualquier LLM los lee. Git los versiona. Funcionan sin internet. No dependen de que una empresa mantenga una plataforma. El sistema sobrevive a que cambies de Claude a Gemini a GPT-4 sin perder nada.
Skills

Cómo funciona un skill

Un skill es un archivo nombre-skill.md con instrucciones precisas para el LLM. Cuando escribes /nuevo-proyecto, el LLM lee ese archivo y lo ejecuta como un programa.

Cuándo se cargan
Solo cuando los invocas. Costo cero hasta que los usas. Un skill de 200 líneas cuesta ~500 tokens — solo en el momento del invocación. El resto del tiempo no existen en tu contexto.
Dónde viven
Claude Code: ~/.claude/skills/*.md
Antigravity: ~/.gemini/antigravity/global_workflows/
Cursor: .cursor/commands/*.md
Todos instalados por bin/install.sh.
Skill Qué hace exactamente Cuándo
/new-session Lee SESSION block (~60t) + fase actual ROADMAP (~100t). Detecta commits sin checkpoint → reconstruye desde git. Genera plan en ≤12 líneas. Cada sesión
/checkpoint Actualiza PROJECT_STATE desde git log. Sincroniza avance ROADMAP. Detecta GO/NO-GO si fase completada. Actualiza ARCHITECTURE/PROBLEMS solo si hay señales. Propone commit. Cada sesión
/new-project Assessment IML (7 preguntas, 6 dimensiones, score Verde/Amarillo/Rojo). Genera SPEC.md, ROADMAP con GO/NO-GO, ARCHITECTURE, PROJECT_STATE. Stack adaptado a constraints reales. Por proyecto
/setup-dupla Entrevista en bloques (identidad, stack, proyectos, plugins). Genera CLAUDE.md + SYSTEM.md + MEMORY_GLOBAL.md. Detecta IDE y llama a install.sh automáticamente. Una vez
/adapt-project Detecta QUICKSTATE.md (micro → full). Lee estructura actual, mapea docs viejos, genera lo que falta. Muestra diff antes de cada cambio. Nunca borra — archiva. Onboarding
/quick-start Crea QUICKSTATE.md con 2 preguntas. Sin docs/, sin git obligatorio. Si >14 días activo, sugiere escalar a /adapt-project. Micro → full sin perder nada. Micro sesiones
/health-check 6 checks: herramientas, global, proyecto, equipo, coherencia stack/código, evaluador (SESSION vs git, hooks en disco, archivos huérfanos). CRITICAL / WARNING / INFO. Mantenimiento
/update-dupla Compara versión local vs GitHub. Backup automático. Re-detecta IDEs. Actualiza skills + hooks en todos los entornos. Propaga templates cambiados a proyectos (aditivo, nunca sobreescribe). Por versión
/knowledge-graph Usa graphify para transformar docs/ en graph.html (visual), graph.json (legible por AI) y GRAPH_REPORT.md. 5× más eficiente en tokens que leer todos los docs por separado. Por fase
/restore Lista save points (git stash o _versions/ si sin git). Permite revertir estado del proyecto a un punto anterior. Si algo falla
/token-budget Avisa cuando el contexto se acerca al límite y propone acciones más livianas. Activo en Claude Code. Automático
Hooks

Automatizaciones fuera del contexto

Los hooks son scripts bash que corren automáticamente en eventos del IDE — sin usar tokens. Solo disponibles en Claude Code (IDE/CLI). En otros entornos, hay que hacerlos manualmente.

suggest-checkpoint.sh
Evento: Stop (cuando Claude termina de responder)
Cuenta commits desde el último checkpoint. Si hay >5 commits sin guardar → avisa. Si hay cualquier commit → avisa cada 3 stops. Usa contador en /tmp/ para no spamear.
auto-snapshot.sh
Evento: Stop
Copia docs/ a _versions/YYYY-MM-DD_HH-MM/ en cada stop. Sin tocar git. Funciona incluso en proyectos sin git. Es el historial de seguridad de tus docs.
guard-project-state.sh
Evento: PreToolUse (Write)
Antes de que Claude escriba cualquier archivo: verifica si el proyecto tiene CLAUDE.md. Si no existe → bloquea el write. Evita que el LLM cree archivos en proyectos no inicializados.
sync-gemini.sh
Evento: PostToolUse (Write)
Después de que Claude escribe ~/.claude/CLAUDE.md: copia automáticamente a ~/.gemini/GEMINI.md. Tu identidad global siempre sincronizada entre Claude y Gemini.
# Hooks configurados en ~/.claude/settings.json
{
  "hooks": {
    "Stop":            ["suggest-checkpoint.sh", "auto-snapshot.sh"],
    "PreToolUse":      { "Write": "guard-project-state.sh" },
    "PostToolUse":     { "Write": "sync-gemini.sh" },
    "UserPromptSubmit": ["session-reminder.sh"]
  }
}
# Todos cross-platform: python3 primary, cksum/date fallback (Windows compatible)
En Cursor, Windsurf y Chat: compensación manual
No hay hooks automáticos fuera de Claude Code. La compensación: ejecutar /checkpoint manualmente al cerrar cada sesión. El sistema no te pierde el estado — pero no te avisa automáticamente.

Claude Code — cómo instalar
git clone https://github.com/leo1z/dupla-workflow
bash dupla-workflow/bin/install.sh
Detecta VS Code, Antigravity y Cursor automáticamente. Instala skills + hooks en una pasada.
Cursor — clave: usa Agent Mode
Skills disponibles en .cursor/commands/. Chat Mode y Composer no leen rules — siempre usa Agent Mode para que los skills funcionen.
⚠️ Chat Mode = no funciona ✅ Agent Mode = OK
Antigravity — skills como texto
Si el skill no se activa con /nombre, simplemente escribe el nombre del skill como texto — Gemini lo lee como instrucción y lo ejecuta igual. GEMINI.md se carga automáticamente.
Web Chat — solo continuidad
Adjunta PROJECT_STATE.md + pega el <handoff> block. Útil para consultas rápidas o cuando estás en movimiento. No intentes construir desde aquí.

Handoff

Cambiar de LLM o entorno sin perder contexto

El <handoff> block es el puente entre entornos. Funciona en absolutamente todos — incluso en WhatsApp o Telegram.

Cómo generar un handoff
1
En el entorno actual: /checkpoint → opción Handoff
2
Copia el bloque <handoff> generado
3
En el nuevo entorno: pega el bloque + escribe /new-session
# Estructura del bloque handoff
<handoff>
Date:    2026-05-03
From:    Claude (sesión 8)
To:      Next session
Project: c:\Users\Leo\Projects\mi-app
Goal:    Implementar auth con Supabase
Branch:  work
Done:    Login UI completo · API endpoint
Next:    Conectar frontend con endpoint /auth
Context: Leer docs/PROJECT_STATE.md primero
</handoff>
Destino Cómo recibir el handoff
Claude Code IDE/CLI Nuevo chat → navega al proyecto → pega <handoff> → escribe /new-session
Antigravity / Gemini Nuevo chat → pega <handoff> → escribe "Ejecuta new-session" (o el nombre del skill)
Cursor Nuevo chat en Agent Mode → pega <handoff> → escribe /new-session
Web Chat / GPT Adjunta PROJECT_STATE.md → pega <handoff> + "Continúa desde Next:"
WhatsApp / Telegram Pega <handoff> block + contenido de claude-progress.txt → "Continúa desde Next:"
sync-gemini: handoff automático para CLAUDE.md
Cuando Claude Code escribe ~/.claude/CLAUDE.md, el hook sync-gemini.sh copia automáticamente a ~/.gemini/GEMINI.md. Tu identidad siempre sincronizada — sin hacer nada manualmente.
Multi-Agente

Trabajo paralelo y patrón de roles

Dupla OS no impone un framework de agentes — usa las capacidades nativas de cada entorno. El patrón recomendado es siempre el mismo.

🗺️
Planificador
Lee PROJECT_STATE + ROADMAP → divide trabajo en tareas independientes con contexto completo
→
⚙️
Implementadores
Cada uno recibe tarea + rama + archivos a tocar. Trabajan aislados.
→
🔍
Evaluador
/health-check verifica coherencia entre lo hecho y el estado declarado antes del merge.
Claude Code — paralelismo nativo
Agent tool con run_in_background: true. Cada subagente recibe contexto completo en su prompt. Los resultados se consolidan al final.
Cursor / Antigravity — paralelismo manual
Abre múltiples chats (tabs) en Agent Mode. Cada tab trabaja en su rama. Consolidas al final con /checkpoint antes del merge.
Regla clave del paralelismo
Cada agente paralelo necesita su propio contexto completo. No asumas que un subagente sabe lo que hizo el otro. Siempre /checkpoint antes del merge para verificar coherencia.
Roadmap

Roadmap dinámico con checkpoints

El ROADMAP no es un documento estático — se actualiza solo en cada /checkpoint cuando el LLM detecta Outcomes completados en el git log.

# Estructura de cada fase en ROADMAP.md
## Phase 1 — MVP

Outcomes:
[x] Login funcional con Supabase         ← marcado por /checkpoint
[ ] Dashboard con datos reales

GO/NO-GO Checkpoint:
  ✅ Continue if: 3+ usuarios activados en la semana
  ⚠️ Adapt if:  usuarios activados pero no retornan → revisar onboarding
  ❌ Kill if:    la asunción "las empresas pagarían $X" resulta falsa
Auto-actualización
/checkpoint compara git log con Outcomes de la fase. Si un Outcome aparece en los commits → lo marca [x] automáticamente.
GO/NO-GO alerta
Si todos los "Continue if" están cumplidos → Phase_Status pasa a GO/NO-GO Pending y /new-session lo muestra como alerta prioritaria.
Kill conditions
Cada fase tiene una condición que mata el proyecto. Las respuestas del IML Assessment alimentan directamente el Kill if de Phase 1.
IML Assessment

Evaluación de madurez de idea

Antes de construir cualquier cosa, /new-project evalúa si la idea vale el esfuerzo. 7 preguntas, 6 dimensiones, 3 salidas posibles.

🟢
Verde — construye
4+ dimensiones fuertes. El sistema lista las fortalezas y el riesgo principal a monitorear. Continúa a entrevista completa.
🟡
Amarillo — con advertencia
2-3 dimensiones fuertes. El sistema detalla qué gaps hay y ofrece: continuar asumiendo riesgos, validar primero, o salir.
🔴
Rojo — valida primero
0-1 dimensiones fuertes. Stop. El sistema da acciones concretas de validación y un prompt para ejecutarlas ahora mismo.
Las 6 dimensiones evaluadas
Problema
¿Específico con momento concreto, o vago y genérico?
Validación
¿3+ personas reales confirmaron, o solo opinión propia?
Disposición a pagar
¿Cifra concreta, o "tal vez / depende"?
Mercado
¿Alternativas claras con gaps, o "no existe nada igual"?
Ventaja real
¿Acceso exclusivo o skill específico, o "soy muy apasionado"?
Riesgo nombrado
¿Asunción específica que si es falsa mata el proyecto?
