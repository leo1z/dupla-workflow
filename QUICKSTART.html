Para nuevos usuarios

Empezar con Dupla Workflow

Guía paso a paso para configurar tu sistema en 15 minutos y tener el primer proyecto en marcha.

Requisitos

Antes de empezar: elige tu IDE

Dupla Workflow funciona en 3 IDEs. Elige UNO e instálalo primero.

⚡
VS Code + Claude Code
Instala: VS Code, luego extensión "Claude Code" en marketplace. Mejor para: desarrolladores que ya usan VS Code.
🔀
Antigravity
Instala: Antigravity (Google's agent IDE). Mejor para: cambiar entre Claude y Gemini sin cerrar la app.
💻
Claude Desktop
Instala: Claude.ai desktop app (Anthropic official). Mejor para: algo simple y directo.
💡 Si no sabes cuál elegir
Recomendado: VS Code + Claude Code. Es lo más usado. O Antigravity si quieres cambiar entre Claude y Gemini en la misma sesión.
Paso 1 de 5

Instalar Dupla Workflow (terminal)

Una sola vez. Abre tu terminal/command line y ejecuta estos comandos:

$ git clone https://github.com/leo1z/dupla-workflow.git $ cd dupla-workflow $ bash bin/install.sh
🖥️ Compatibilidad
macOS: Terminal nativo — funciona directo.
Windows: Usa Git Bash (no PowerShell). En VS Code: terminal → ∨ → Git Bash.
Linux: Terminal nativo — funciona directo.
¿Qué hace bin/install.sh?
Detecta: ¿tienes VS Code? ¿Antigravity? ¿Claude Desktop?
Instala: 13 skills en ~/.claude/skills/ (+ copia a ~/.agent/skills/ si tienes Antigravity).
Despliega: 3 hooks (automaciones que corren fuera del contexto — sin quemar tokens).
Resultado: El sistema está listo. Después no tocas más la terminal hasta /new-project.
Paso 2 de 5

Configurar tu máquina (IDE)

Una sola vez, en tu IDE. Si ya tienes CLAUDE.md o SYSTEM.md, el sistema lo detecta y pregunta si actualizar.

1
Abre tu IDE
VS Code + Claude Code, Antigravity, o Claude Desktop. En cualquier carpeta (puede ser tu Downloads). Si en Antigravity, puedes elegir Claude o Gemini aquí — cualquiera funciona.
2
Escribe el comando (IDE input)
En el input de chat (abajo), escribe:
/setup-dupla
3
Responde preguntas (si las hay)
Si es primera vez: Preguntas agrupadas (quién eres, stack, proyectos, plugins). Responde todo de una vez.
Si ya existe CLAUDE.md/SYSTEM.md: El sistema pregunta "Encontré CLAUDE.md, ¿actualizar? [s/n]". Elige sí o no para cada archivo.
4
Confirmación
El sistema genera o actualiza: ~/.claude/CLAUDE.md (identidad), ~/.claude/SYSTEM.md (stack + proyectos), ~/.claude/PROBLEMS_GLOBAL.md (aprendizajes). Listo.
✅ Listo — tu máquina ya conoce quién eres
Los archivos generados se leen automáticamente en cada proyecto. Puedes editarlos si algo cambia (stack nuevo, proyecto nuevo, etc.).
Importante

Multi-IDE y cambiar de modelo (Claude ↔ Gemini)

Funciona en Antigravity, Claude Desktop, VS Code. Puedes cambiar de LLM entre sesiones.

3 IDEs soportados · macOS / Windows / Linux
Antigravity: Claude + Gemini en el mismo lugar. Claude Desktop: app nativa de Anthropic. VS Code: extensión Claude Code. Elige uno. El sistema funciona igual en los tres y en los 3 sistemas operativos.
Skills en Antigravity
install.sh los instala automáticamente en ~/.agent/skills/. En Antigravity los skills se activan escribiendo su nombre (ej: "Ejecuta /new-session") — no como slash commands como en Claude Code.
💡 Handoff Claude ↔ Gemini
Para pasar de Claude a Gemini: /checkpoint handoff → elige Gemini → genera bloque <handoff> con instrucciones exactas. En Antigravity: nuevo chat → pega el bloque → escribe "Lee el bloque handoff y corre /new-session". El modelo receptor sabe exactamente dónde retomar.
Paso 3 de 5

Crear tu primer proyecto

Tres modos según el tamaño del proyecto. Dupla Workflow se adapta a dónde estés.

Proyecto nuevo
Idea fresca. /new-project te pregunta: ¿problema, usuario, MVP, riesgos? Genera docs + carpetas automáticamente. Puedes empezar desde cualquier punto de madurez (idea nueva, ya investigada, MVP definido).
Proyecto existente
Ya tienes código, docs, historia. /adapt-project lee tu estructura actual, mapea docs viejos, genera lo que falta. Nada se borra, todo se integra.
⚡ Sesión rápida (Micro)
Script, experimento, tarea no-código. /quick-start — 2 preguntas, crea un QUICKSTATE.md (~80 tokens), listo. Sin carpeta docs/, sin git, sin roadmap. Perfecto para sesiones que no necesitan estructura completa.
1
Crea una carpeta nueva
En tu terminal:
mkdir mi-proyecto && cd mi-proyecto && git init
2
Abre Claude Code en esa carpeta
Navega a mi-proyecto/ en Claude Code.
3
¿De dónde estás partiendo?
Idea nueva: cuéntale todo (problema, usuarios, MVP, riesgos, restricciones). Ya investigada: menciona dónde está tu research. MVP definido: comparte dónde está tu roadmap. /new-project se adapta y salta pasos que ya hiciste.
4
Corre el comando de inicialización
Una vez que ya explicaste el proyecto:
/new-project
5
Revisa lo que se generó
Dupla Workflow crea automáticamente: CLAUDE.md (contexto del proyecto), docs/PROJECT_STATE.md (estado actual), docs/ROADMAP.md (plan), docs/ARCHITECTURE.md (decisiones técnicas).
✅ Tu proyecto está inicializado
Ahora tienes una estructura clara. En cada sesión, el sistema sabe dónde estás sin que tengas que re-explicar.
Paso 4 de 5

Primera sesión de trabajo

Cada vez que abras Claude Code en tu proyecto, este es el flujo.

1
Iniciar sesión
Escribe:
/new-session
Claude leerá automáticamente dónde quedaste. Te dirá: qué hacer, con cuál LLM trabajar, qué está bloqueando.
2
Trabajar
Haz lo que el sistema te indicó. Edita archivos, ejecuta comandos, resuelve problemas.
3
Guardar progreso (/checkpoint)
Antes de cerrar la sesión o cambiar de modelo:
/checkpoint
Qué hace:
1. Quick save (mid-sesión): "¿Qué hiciste?" → actualiza PROJECT_STATE.md
2. Close (end of day): idem + pregunta por resolver problemas → archiva en PROBLEMS.md
3. Handoff (cambiar LLM): idem + genera bloque de traspaso + propone push

En todos los casos: propone mensaje de commit + recomienda próximo modelo.
4
Hacer commit y push
Git push. Listo. La próxima sesión sabrá exactamente dónde quedaste.
✅ Sesión guardada
PROJECT_STATE.md es tu "memoria": qué se hizo, qué sigue, qué está bloqueando. Sin esto, cada sesión sería como empezar de cero.
Paso 5 de 5

Mantener el sistema actualizado

Ocasionalmente, revisa que todo esté coherente. Es rápido.

Cada 2-4 semanas
Corre /health-check para verificar que todos los docs del proyecto estén actualizados, el stack coincida con el código, y los skills estén bien instalados.
Mantenimiento
Cuando haya nueva versión
Corre /update-dupla desde tu IDE, o en terminal: cd ~/dupla-workflow && git pull && bash bin/install.sh. Hace backup automático antes de actualizar.
Según necesidad
Referencia

Los 4 archivos que todo proyecto tiene

Después de /new-project, cada proyecto se ve así:

📋
PROJECT_STATE.md
El más importante
Tu "fuente de verdad". Dice: dónde estás ahora, qué estás haciendo, qué sigue, qué te bloquea. Se actualiza con /checkpoint cada sesión. Lee esto primero cada mañana.
## Status Updated: 2026-04-21 Next: Implementar login con Google
🗺️
ROADMAP.md
Plan completo
Divide el proyecto en fases (2 semanas cada una). Cada fase tiene un GO/NO-GO: condiciones para continuar. Te dice si estás en camino o necesitas ajustar.
🏗️
ARCHITECTURE.md
Decisiones técnicas
Stack elegido, diagramas, decisiones de diseño. "¿Por qué usamos Supabase en lugar de Firebase?" Respuestas acá. Para cuando necesites refactorar o agregar un nuevo dev.
⚙️
CLAUDE.md
Contexto del proyecto
Lo que Claude necesita saber para trabajar en este proyecto: cómo el stack se conecta, dónde viven los archivos críticos, reglas especiales (no tocar la base de datos en esta etapa), etc.
Comprensión

¿Cómo funciona sin que yo tenga que pedirlo?

Cada comando de Dupla Workflow lee archivos automáticamente. No necesitas decir "lee PROJECT_STATE".

🤖
/new-session
Lee SESSION block (~60t) + fase actual del ROADMAP (~100t) → detecta si hubo commits sin checkpoint → te dice dónde estás y qué hacer.
💾
/checkpoint
Actualiza PROJECT_STATE + sincroniza avance del ROADMAP + detecta si completaste una fase (GO/NO-GO). Solo pregunta sobre arquitectura o problemas si hubo cambios relevantes.
🔍
/health-check
Revisa skills instalados, docs del proyecto, SESSION block actualizado y coherencia del stack. Dice exactamente qué arreglar si algo está mal.
🆕
/new-project
Crea ROADMAP con fases + GO/NO-GO criteria, ARCHITECTURE.md, y PROJECT_STATE con la fase activa marcada desde el inicio.
Analogía
Dupla Workflow es como tener un asistente que sabe exactamente dónde archivas todo. Cuando dices "necesito trabajar en el login", el asistente sabe dónde está el código, qué se hizo antes, y qué sigue — sin que tengas que explicarlo cada vez.
Workflow

Individual vs Equipo

Mismo sistema, flujo adaptado. Defines el modo cuando creas el proyecto — y puedes cambiar después.

👤
Individual
Flujo diario:
1. /new-session → lee dónde quedaste (~60 tokens)
2. Trabaja en tu rama
3. /checkpoint → guarda estado (auto-detecta qué hiciste del git log)
4. git push

PROJECT_STATE.md: 1 bloque <session> que /checkpoint actualiza automáticamente.

Cambiar LLM: /checkpoint → opción 3 (Handoff) → genera bloque de traspaso → nuevo chat.
👥
Equipo (2-4 personas)
Flujo diario (Dev):
1. /new-session → lee TU sección + dependencias (~60 tokens total)
2. Trabaja en tu rama (work/fase-rol)
3. /checkpoint → actualiza TU sección (auto del git log)
4. Push a tu rama

Flujo Lead:
/new-session standup → tabla consolidada del equipo (quién hizo qué, bloqueados, PRs pendientes).
/checkpoint approve-pr work/[rama] → ve diff + ROADMAP → verifica SESSION del dev → aprueba → mergea a main.
🌿 Estrategia de ramas (sugerida automáticamente)
# Estructura generada automáticamente en /new-project (Team)
main ← producción (Lead aprueba)
├─ work/phase1-backend ← Dev A
├─ work/phase1-frontend ← Dev B
└─ work/phase1-db ← Dev C (empezar aquí)
Cada dev en su propia rama = sin conflictos. El sistema dice en qué orden hacer merge según las dependencias del ROADMAP.
📋 PROJECT_STATE.md en equipo (secciones por dev)
## Team Status

### Dev A (Backend) — work/phase1-backend
Updated: 2026-04-21
Done: auto del git log
Next: Implementar login endpoint
Dependencies: Espera Dev C (DB schema)

### Dev B (Frontend) — work/phase1-frontend
Done: Login form UI
Next: Integración con API
Dependencies: Espera Dev A (API lista)

## Shared Status (Lead only)
Last Merge: Dev C — DB schema — hoy
/new-session lee solo tu sección (~30 tokens). Done se genera automáticamente del git log — sin escribir nada. Dos devs pueden pushear al mismo tiempo sin conflicto porque cada uno tiene su sección.
💡 ¿Cómo empieza un equipo?
Proyecto nuevo: En /new-project, elige "Equipo" → pregunta quién es Lead + miembros + roles → genera todo (docs, ramas, estrategia git).

Proyecto individual ya existente: /adapt-to-team → hace backup → agrega Team sections a docs existentes → crea ramas de trabajo.

Agregar dev a mitad del proyecto: /add-team-member → actualiza todos los docs + crea su rama.

Referencia

Todos los comandos

13 skills en total. La mayoría no los necesitas a diario — los dos que usas todo el tiempo son /new-session y /checkpoint.

Comando Cuándo usarlo Frecuencia
/new-session Al abrir un chat nuevo o empezar a trabajar. Lee el estado del proyecto y genera el plan del día. Cada sesión
/checkpoint Al cerrar la sesión, a mitad del día, o antes de cambiar de modelo (Claude ↔ Gemini). Guarda estado, sincroniza ROADMAP, propone commit. Cada sesión
/setup-dupla Una sola vez por máquina. Configura tu identidad, stack y proyectos en ~/.claude/CLAUDE.md. Una vez
/new-project Para inicializar un proyecto nuevo desde cero. Hace el assessment IML, genera ROADMAP con GO/NO-GO, ARCHITECTURE y PROJECT_STATE. Por proyecto
/adapt-project Para integrar un proyecto existente o para actualizar CLAUDE.md cuando el stack o contexto del proyecto cambia significativamente. Nunca borra — muestra diff y pide aprobación. Onboarding · Refresh
/quick-start Para scripts, experimentos o sesiones pequeñas. Crea un QUICKSTATE.md. /quick-start save o /quick-start close guarda y cierra la sesión micro explícitamente. Sesiones micro
/adapt-to-team Convierte un proyecto individual a modo equipo. Agrega secciones por dev, crea ramas de trabajo. Hace backup automático. Según necesidad
/add-team-member Agrega un nuevo dev a un proyecto equipo ya iniciado. Actualiza PROJECT_STATE, ROADMAP y crea su branch. Según necesidad
/health-check Verifica que el sistema esté bien: skills instalados, docs actualizados, SESSION no stale, stack coherente. Dice exactamente qué corregir. Cada 2-4 semanas
/update-dupla Actualiza el sistema a la última versión. Muestra changelog, hace backup automático y actualiza los 13 skills. Por versión
/restore Muestra puntos de guardado anteriores (save points) y permite revertir el estado del proyecto a uno de ellos. Si algo sale mal
/project-audit Audita el impacto de cambios usando el code-review-graph. Útil antes de refactors grandes o al iniciar una nueva fase. Por fase
/token-budget Avisa cuando el contexto del chat está cerca del límite y propone alternativas más livianas para la siguiente acción. Automático
FAQ

Preguntas frecuentes

¿Y si tengo un proyecto que ya existe?
Corre /adapt-project en lugar de /new-project. El sistema detecta qué docs faltan y genera solo lo necesario — sin pisar lo que ya existe.

También úsalo cuando el proyecto evoluciona: si cambias el stack, agregas una dependencia importante, o el CLAUDE.md ya no refleja el estado real del proyecto — /adapt-project detecta el drift y propone actualizaciones con diff y aprobación.
¿Para qué sirve /quick-start?
Para scripts, experimentos, research o tareas pequeñas donde no necesitas un proyecto completo. Solo crea un archivo QUICKSTATE.md con qué es, qué hiciste y qué sigue — 80 tokens totales. Si el proyecto crece, puedes migrarlo con /adapt-project.
¿Cuánto se tarda en /new-project?
2-3 minutos máximo. El sistema genera 4 archivos. Luego puedes editarlos si quieres ser más específico.
¿Puedo usar Gemini en lugar de Claude?
Sí. Claude se usa para implementar, debug y decisiones de arquitectura. Gemini para research, análisis y code review de features aisladas. /checkpoint te recomienda cuál modelo usar al cerrar cada sesión — no tienes que decidirlo tú.
¿Qué pasa si no corro /checkpoint?
Si hay commits nuevos, /new-session detecta automáticamente que hubo cambios y reconstruye el estado del SESSION block desde git. Sin embargo, pierde la sincronización del ROADMAP: si completaste una fase sin hacer checkpoint, no se marca como hecha. /checkpoint toma 30 segundos y evita ese drift.
¿Los archivos se pushen a Git?
Sí, PROJECT_STATE.md, ROADMAP.md, y otros se pushean. Así tu equipo (o tu yo del futuro) ve el estado. CREDENTIALS.md nunca se pushea — está en .gitignore.
¿Qué es MCP?
Opcional. File system access para Claude (solo en Claude Desktop/Code). /setup-dupla te pregunta si quieres habilitarlo. Si no sabes qué es, elige "no". No lo necesitas para usar Dupla Workflow.
¿Puedo cambiar de Gemini a Claude durante un proyecto?
Sí, en Antigravity. /checkpoint te recomienda cuál modelo es mejor para lo siguiente. Si quieres cambiar: /checkpoint, cambia de LLM en Antigravity, luego /new-session en el nuevo modelo.
Mi idea ya está investigada, ¿qué hago?
/new-project lo detecta en Phase 0. Responde cuál es tu estado (idea nueva, investigada, MVP definido). El sistema salta pasos que ya completaste.
¿Funciona en equipo?
Sí. En /new-project elige "Equipo". El sistema genera ramas por dev, secciones individuales en PROJECT_STATE.md, y estrategia git. Lead usa /checkpoint approve-pr para revisar y mergear. Ver sección "Individual vs Equipo" arriba.
¿Cómo convierto mi proyecto individual a equipo?
/adapt-to-team — hace backup automático, agrega Team sections a todos los docs existentes, crea ramas de trabajo. Nada se pierde.
¿Cómo agrego un nuevo dev a mitad del proyecto?
/add-team-member — agrega su sección en PROJECT_STATE, actualiza CLAUDE.md y ROADMAP, crea su rama. También sirve para cambiar roles.
¿Cuándo abro un nuevo chat?
Cuando llevas 25–30 mensajes en el mismo chat. El contexto crece, Claude se vuelve más lento. Flujo: /checkpoint → cierra ese chat → abre uno nuevo → /new-session. El sistema sabe exactamente dónde quedaste. El hook de Claude te avisará automáticamente cuando detecte sesión larga.
¿Cómo actualizo Dupla-Workflow?
Desde IDE: /update-dupla — detecta tu versión, muestra changelog, hace backup automático, actualiza. Desde terminal: cd ~/dupla-workflow && git pull && bash bin/install.sh. Si venías de v1: el mismo comando detecta v1 y guía la migración.
¿El sistema verifica si mis docs están actualizados?
/health-check — verifica: skills instalados, docs del proyecto, PROJECT_STATE no stale, stack en CLAUDE.md coherente con código, CREDENTIALS.md en .gitignore, versión actual. Si algo está mal, te dice exactamente qué comando lo arregla.
